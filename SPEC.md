
# Family QT Web App Spec

## 1. 제품 개요
가족끼리 매일 QT(Quiet Time) 나눔을 기록하고, 서로의 나눔에 답글을 남길 수 있는 비공개 모바일 우선 웹앱을 만든다.

이 앱은 SNS처럼 시끄럽거나 경쟁적인 분위기가 아니라, 차분하고 따뜻한 “가족 묵상실” 같은 경험을 목표로 한다.

기본 전제:
- 모바일 우선
- 비공개 가족용 앱
- 오늘의 본문은 성경 본문 텍스트가 아니라 **관리자가 수기로 입력한 본문 reference 문자열**만 표시
- 예: `시편 24:1-7`
- 각 사용자는 **자기 현지 날짜(local date)** 기준으로 “오늘”과 “오늘의 본문”을 본다

기본 UI 언어:
- 한국어

---

## 2. 제품 목표

### 꼭 필요한 목표
1. 각 가족 구성원은 자기 현지 날짜 기준으로 오늘의 본문 reference를 볼 수 있어야 한다.
2. 각 가족 구성원은 하루에 자기 나눔을 작성할 수 있어야 한다.
3. 가족 구성원은 서로의 나눔을 읽고 답글을 달 수 있어야 한다.
4. 관리자는 날짜별 본문 reference를 수기로 입력/수정할 수 있어야 한다.
5. 기록이 쌓이면서 과거 나눔을 다시 찾아볼 수 있어야 한다.

### 의도적으로 하지 않는 것 (V1)
- 성경 본문 원문 자동 불러오기
- 공개 공유
- 과도한 SNS 기능
- 좋아요 수 경쟁
- 뱃지, 순위, 연속 기록(streak) 같은 게임화
- 파일 업로드
- 리치 텍스트 에디터
- 푸시 알림
- 여러 가족 그룹을 지원하는 범용 서비스화

---

## 3. 사용자와 역할

### Member
- 가족 구성원
- 오늘의 본문 보기
- 자기 나눔 작성 / 수정 / 삭제
- 다른 가족 구성원의 나눔 읽기
- 답글 작성 / 수정 / 삭제
- 기록 보기

### Admin
- Member 권한 전부 포함
- 날짜별 본문 reference 입력 / 수정 / 삭제
- 추후 필요 시 관리 기능 확장 가능

V1에서는 **하나의 가족만 존재하는 단일 가족 앱**으로 가정한다.
즉, 처음부터 범용 멀티테넌트 제품처럼 복잡하게 만들지 않는다.

## 4. 핵심 규칙

### 4.1 오늘의 기준
- “오늘”은 각 사용자의 **IANA timezone** 기준으로 계산한다.
- 예:
  - `Asia/Seoul`
  - `America/New_York`
- 절대로 `UTC+9`, `UTC-5` 같은 고정 오프셋만으로 처리하지 않는다.
- DST(서머타임) 때문에 고정 오프셋은 사용하지 않는다.

### 4.2 오늘의 본문 기준
- 각 사용자는 자기 **현지 날짜(local date)** 기준으로 오늘의 본문을 본다.
- 예:
  - 사용자 A의 local date가 `2026-03-22`이면 `2026-03-22`의 본문을 본다.
  - 사용자 B의 local date가 `2026-03-23`이면 `2026-03-23`의 본문을 본다.
- 즉, 같은 절대 시각이어도 서로 다른 날짜의 본문을 볼 수 있다.

### 4.3 본문 입력 방식
- 관리자는 날짜별로 본문 reference 문자열만 수기로 입력한다.
- 예:
  - `시편 24:1-7`
  - `요한복음 3:16-21`
- 외부 Bible API 연동은 하지 않는다.
- 성경 본문 텍스트를 직접 저장하거나 표시하지 않는다.

### 4.4 나눔 작성 규칙
- 각 사용자는 **local_date당 게시된 나눔 1개**만 가진다.
- Draft(임시저장)는 존재할 수 있다.
- Draft는 특정 local_date에 귀속된다.
- Draft를 작성하다 자정을 넘겨도, 기존 draft는 원래 local_date를 유지한다.
- 새로운 날짜의 글을 쓰고 싶으면 새로 시작해야 한다.

### 4.5 본문 snapshot 규칙
- 사용자가 나눔을 저장/게시할 때, 그 시점의 본문 reference를 엔트리에 snapshot으로 저장한다.
- 나중에 관리자가 날짜별 본문 reference를 수정하더라도, 이미 작성된 과거 엔트리의 본문 표시는 바뀌지 않는다.

### 4.6 답글 규칙
- 답글 개수 제한은 없다.
- V1에서는 **thread depth = 1** 로 한다.
- 즉:
  - 게시글에 답글은 무제한 가능
  - 답글에 다시 답글을 다는 중첩 구조는 V1에서 지원하지 않는다
- 답글이 많아질 경우 UI는 접기/펼치기 또는 “답글 더 보기” 방식으로 처리한다.

### 4.7 시간 저장 규칙
- 모든 실제 시각(timestamp)은 UTC로 저장한다.
- 동시에 조회/정렬/아카이브를 위해 `local_date`를 별도 저장한다.
- “created_at”과 “local_date”는 서로 다른 용도다.
- 절대로 timestamp만으로 local date를 매번 추정하지 않는다.

### 4.8 접근 제어
- 이 앱은 비공개 가족 앱이다.
- 로그인하지 않은 사용자는 앱 내용을 볼 수 없다.
- Admin 페이지는 관리자만 접근 가능해야 한다.

## 5. V1 화면 목록

### 5.1 로그인 / 진입
경로 예시:
- `/login`

목표:
- 이메일 기반 간단 로그인
- 비공개 앱 진입

권장:
- 매직 링크 또는 간단한 이메일 인증 방식

---

### 5.2 홈
경로 예시:
- `/`

핵심 목적:
- 사용자가 앱에 들어오자마자 오늘 해야 할 일과 가족의 오늘 나눔을 볼 수 있어야 한다.

구성 요소:
1. 상단 헤더
   - 앱 이름
   - 오늘 날짜 (사용자 local date 기준)
   - 설정 또는 프로필 진입

2. 오늘의 본문 카드
   - 라벨: `오늘의 본문`
   - 본문 reference 표시
   - 예: `시편 24:1-7`

3. 가족 상태 영역
   - 가족 구성원별 상태 표시
   - 예:
     - `완료`
     - `작성중`
     - `아직`
   - 경쟁 느낌 없이 담백하게 표시

4. 오늘 나눔 CTA
   - `오늘 나눔 쓰기`
   - 오늘 draft가 있으면 `이어쓰기`
   - 오늘 게시한 글이 있으면 `내 글 보기`

5. 오늘의 가족 피드
   - 사용자 local date 기준 “오늘” 작성된 가족 나눔 목록
   - 카드 형태
   - 카드 클릭 시 상세 화면 이동

UI 원칙:
- 모바일 우선
- 차분한 카드형 레이아웃
- 텍스트 읽기 쉬움
- 과한 배지/카운트 지양

---

### 5.3 나눔 작성
경로 예시:
- `/write`

핵심 목적:
- 사용자가 부담 없이 오늘의 나눔을 기록할 수 있어야 한다.

필드:
1. 오늘의 본문 reference
   - 읽기 전용 표시
   - 예: `시편 24:1-7`

2. 마음에 남은 한 줄
   - optional
   - 짧은 텍스트

3. 오늘의 나눔
   - required
   - multiline
   - 가장 핵심 필드

4. 적용 / 기도제목
   - optional
   - multiline

버튼:
- `임시저장`
- `등록하기`

행동 규칙:
- autosave가 있으면 좋다
- draft가 있으면 이어서 편집 가능
- 게시 시 본문 reference snapshot 저장

폼 UX:
- 모바일에서 입력이 쉬워야 한다
- placeholder는 구체적이고 부드럽게
- 큰 입력창과 넉넉한 여백
- 하단 CTA가 명확해야 한다

---

### 5.4 나눔 상세
경로 예시:
- `/entries/[id]`

핵심 목적:
- 하나의 나눔을 깊이 읽고, 답글로 대화를 이어간다.

구성 요소:
1. 작성자 정보
2. local date
3. 본문 reference snapshot
4. 본문 내용(마음에 남은 한 줄 / 오늘의 나눔 / 적용·기도제목)
5. 답글 목록
6. 답글 작성 폼

답글 UI 원칙:
- 답글 수는 무제한
- V1에서는 게시글 아래 1단계 답글만
- 답글이 많으면 일부만 먼저 보여주고 `답글 더 보기` 지원 가능
- 모바일에서 답글 영역이 과도하게 길어지지 않게 설계

정렬:
- 기본은 오래된 답글 → 최신 답글 순서 추천
- 최신 답글이 아래에 붙는 전통적인 대화 흐름

---

### 5.5 기록
경로 예시:
- `/archive`

핵심 목적:
- 과거 나눔을 날짜별로 다시 볼 수 있어야 한다.

V1 권장 방식:
- 우선은 **날짜 그룹 리스트**로 시작해도 된다
- 가능하면 추후 월별 달력형 UI 추가
- 너무 복잡한 달력 인터랙션으로 시작하지 않는다

필수 요구:
- 날짜별로 과거 엔트리 보기
- 각 날짜의 본문 reference snapshot과 작성자 확인 가능
- 과거 나눔 상세로 이동 가능

선택 요구:
- 가족 구성원 필터

---

### 5.6 관리자 본문 관리
경로 예시:
- `/admin/passages`

핵심 목적:
- 관리자가 날짜별 본문 reference를 입력/수정/삭제할 수 있어야 한다.

필드:
- 날짜
- 본문 reference 문자열

예:
- 날짜: `2026-03-22`
- 본문: `시편 24:1-7`

필수 기능:
- 추가
- 수정
- 삭제
- 최근/다가오는 일정 목록 확인

중요:
- 성경 원문 입력 기능은 없음
- reference 문자열만 관리

## 9. 데이터 모델 초안

아래는 구현을 위한 권장 모델이며, 실제 컬럼명은 구현 시 약간 조정 가능하다.

### users
- `id`
- `email`
- `display_name`
- `timezone` (IANA timezone string)
- `role` (`member` | `admin`)
- `created_at`
- `updated_at`

### passage_schedule
- `id`
- `local_date` (YYYY-MM-DD)
- `reference`
- `note` (optional)
- `created_at`
- `updated_at`
- `updated_by`

제약:
- `local_date`는 unique 권장

### entries
- `id`
- `user_id`
- `local_date` (YYYY-MM-DD)
- `passage_reference_snapshot`
- `memorable_line` (optional)
- `reflection` (required)
- `application_or_prayer` (optional)
- `status` (`draft` | `published`)
- `created_at`
- `updated_at`
- `published_at` (optional)

제약:
- user별 published entry는 `local_date`당 1개

### replies
- `id`
- `entry_id`
- `user_id`
- `body`
- `created_at`
- `updated_at`
- `deleted_at` (optional soft delete)

V1 규칙:
- replies는 entry에 직접 달린다
- reply-to-reply 구조는 없음
